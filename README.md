# Task Tracker — фронтенд (этап 1: Auth, сессии, профиль)

React 19 + TypeScript + Vite, AntD (тема: тёплый тёмно-синий, крупные скругления),
Zustand для стейта, SSE для realtime-уведомлений.

## Запуск

```bash
npm install
cp .env.example .env   # укажи реальный VITE_API_URL бэкенда
npm run dev
```

`npm run build` — продакшен-сборка, `npm run preview` — посмотреть собранный билд.

## Что реализовано

- **Регистрация / вход** (`/register`, `/login`) — `AuthDto`-формы, HttpOnly JWT-кука
  ставится бэкендом, фронт её не трогает (`axios` с `withCredentials: true`).
- **Профиль** (`/profile`):
  - загрузка/смена аватарки (PNG/JPEG/WEBP, до 5 МБ, превью сразу обновляется);
  - список активных сессий (устройство по User-Agent, IP, город, дата входа),
    завершение чужой сессии одной кнопкой;
  - удаление аккаунта с подтверждением вводом логина.
- **Realtime**: постоянное SSE-соединение с `/notifications/user` пока пользователь
  залогинен (с переподключением при обрыве), всплывающие уведомления + колокольчик
  с историей последних событий. Отдельный хук `useTaskNotifications` уже готов для
  будущего экрана задачи (`/notifications/task/:id`).
- **Пространства** (`/workspaces`):
  - сетка карточек (личное пространство помечено замком, у остальных — код и роль
    текущего пользователя, если бэкенд её отдаёт);
  - создание пространства (название, короткий код для Jira-нумерации, приватность);
  - Drawer с деталями: редактирование имени/кода (PATCH), список участников с
    ролями, добавление через LIKE-поиск по логину (`GET /users?login=`) с выбором
    роли, удаление участника, удаление пространства целиком;
  - управление видно только если роль пользователя в пространстве — `ADMIN`
    (или роль неизвестна — тогда сервер сам отклонит запрос с 403);
  - приход `WORKSPACE_INVITE` по SSE сразу обновляет список пространств.
- **Задачи** (`/workspaces/:workspaceId` — открывается кликом по карточке
  пространства):
  - Канбан-доска из трёх колонок (TODO / IN_PROGRESS / DONE), карточка
    показывает Jira-код (`WEB-12`), приоритет, исполнителя, срок;
  - перетаскивание карточки между колонками (нативный HTML5 drag-and-drop,
    без сторонних библиотек) меняет статус — это единственное, что разрешено
    роли EXECUTOR;
  - создание задачи (название, описание, приоритет, срок, исполнитель из
    участников пространства);
  - Drawer с деталями задачи: для ADMIN/EDITOR — полное редактирование всех
    полей и удаление; для EXECUTOR — только смена статуса, остальные поля
    задизейблены на фронте (и всё равно проверяются сервером); для OBSERVER —
    доска и карточки полностью read-only, кнопка создания скрыта;
  - приход `TASK_ASSIGNED` по SSE обновляет открытую доску без перезагрузки.
- **Комментарии** (внутри Drawer задачи):
  - дерево комментариев с неограниченной вложенностью (визуальный отступ
    ограничен 3 уровнями, данные — нет); загружаются по
    `GET /tasks/:id/comments/tree`;
  - ответ на комментарий — inline-форма прямо под узлом, без модалок;
  - редактирование своего комментария (Escape — отменить, Ctrl+Enter — сохранить);
  - удаление: автор удаляет свой, ADMIN/EDITOR — любой (с предупреждением о
    каскадном удалении ветки, если у комментария есть ответы);
  - OBSERVER видит комментарии, но форма ввода скрыта;
  - SSE-канал `/notifications/task/:id` подписан, пока Drawer открыт — новые
    комментарии от других участников появляются автоматически без перезагрузки;
    `onEvent`-колбэк стабилизирован через `ref`, поэтому SSE не пересоздаётся
    на каждый рендер.
- Защищённые роуты (`ProtectedRoute`) с проверкой сессии при загрузке приложения,
  стартовый экран после логина — список пространств.

## Важные допущения (нет в Swagger-доке, проверь на бэке)

Спецификация не описывает форму ответов для `User` и `Session` — формы в
`src/api/types.ts` (`AuthUser`, `Session`) собраны по логике ТЗ. Если бэкенд
возвращает другие имена полей — править нужно только этот файл и `src/api/auth.ts`:

- `POST /auth/login` и `/auth/register` — ожидается, что в ответе придёт объект
  пользователя (`{ id, login }`). Если нет — на фронте `id` пользователя останется
  пустым, и аватарку показать не получится, пока не появится способ узнать `id`
  (например, отдельный `GET /auth/me`).
- `GET /auth/sessions` — ожидается массив объектов `{ id, ip, userAgent, city,
  createdAt, isCurrent }`.
- `GET /users/:id/avatar` — отдаёт сам файл картинки (используется как `src` у
  `<img>`/`Avatar`).
- `GET /workspaces` — в свагере схема ответа пустая (`[{}]`), форма объекта
  `Workspace` (поля `id/name/code/isPrivate/members`) — предположение по логике ТЗ.
  Особенно неочевидно, отдаёт ли бэкенд `members` сразу в списке — если нет,
  страница пространств всё равно работает, просто роль участников и счётчик
  команды будут пустыми, пока не появится способ их получить (например,
  `members` в ответе `GET /workspaces` или отдельный роут).
- `POST /workspaces/:id/members` и `DELETE /workspaces/:id/members/:userId` не
  возвращают тело ответа — список участников на фронте обновляется локально
  (оптимистично), а `login` участника берётся из результата LIKE-поиска.
- `Task.status`/`priority` — ровно перечисления из ТЗ (`TODO/IN_PROGRESS/DONE`,
  `LOW/MEDIUM/HIGH/CRITICAL`). `assigneeId`/`dueDate` опциональны.
- У `/notifications/task/:id` пока нет подписчика на доске — канал предусмотрен
  (`useTaskNotifications`) для будущего экрана конкретной задачи с
  комментариями; обновление списка задач при чужих правках сейчас приходит
  только через `TASK_ASSIGNED` на канале пользователя, не на каждое движение
  по доске другим участником. Если нужна live-синхронизация доски между всеми
  участниками — это или новый канал на уровне пространства, или подписка на
  все видимые задачи разом, обсудим при необходимости.
- Формат SSE-сообщений — JSON-конверт `{ type, message?, payload? }`. Если бэкенд
  шлёт что-то другое, поправь `parseEvent` в `src/hooks/useUserNotifications.ts`.

Раз эндпоинта `GET /auth/me` нет, "жива ли сессия" при перезагрузке страницы
фронт проверяет по факту успешного `GET /auth/sessions` (401 → экран логина).

## Сеть

Из песочницы, где собирался проект, нет доступа к серверу `109.68.213.17` —
интеграция с реальным бэкендом не тестировалась вживую. Структура запросов
строго следует Swagger-путям из ТЗ; если при реальном подключении что-то не
сойдётся по форме ответа — смотри раздел допущений выше.

## Статус

Все модули из ТЗ реализованы: Auth + сессии + профиль → Workspaces + роли →
Tasks с Jira-нумерацией → Comments с деревом и realtime. Проект готов к
интеграции с бэкендом.
